# ADR 0001 — 무이주 모듈 경계와 성능 선불 설계

- 상태: 승인
- 날짜: 2026-07-23

## 결정

1. TypeScript strict, pnpm workspace, React/Vite/Tailwind/shadcn 구조를 첫 커밋부터 사용한다.
2. 카드·추출·판정·저장 계약은 React/DOM 없이 Node에서 실행된다. UI는 `packages/ui`,
   `cabinets/*/react`, `apps/web` 끝단에만 존재한다.
3. 카드 파일은 `BinarySource` 구간 읽기를 정본으로 삼고 에셋 바이트를 공용 DTO에 넣지 않는다.
4. 캐비닛은 `core`와 `react`를 분리하고 선언형 매니페스트로 등록한다.
5. 저장은 작은 행동 WAL과 큰 스냅샷·에셋 차선을 분리한다. 현재 A1은 포트까지만 정의하며
   브라우저 구현은 첫 캐비닛 UI 전에 완료한다.
6. 버튼·분석·에셋 성능은 기능 완료 조건이다. 경계 검사, 50/1,000행동 하니스,
   초기 JS gzip 200KiB 상한을 CI에서 검사한다.

## 이유

럭키 시뮬레이터에서는 과거 전체 저장·검증, 원본 이미지 일괄 디코드, 긴 목록 전체 렌더가
버튼 반응을 막았다. 엔진 판정 자체보다 저장과 렌더 차선 결합이 병목이었다. 럭키★오락실은
같은 수술을 반복하지 않도록 입력·판정·WAL·렌더·백그라운드 저장을 처음부터 분리한다.

## 결과

- 프레임워크나 저장 구현을 바꿔도 순수 게임 규칙과 보고서 코드는 유지된다.
- 큰 카드와 에셋 모듈은 중앙 디렉터리와 메타데이터만 먼저 읽을 수 있다.
- 새 캐비닛은 앱 셸 조건문 대신 독립 폴더와 매니페스트로 추가한다.
- 일상적인 의존성 버전 업데이트는 필요하지만 제품 전체를 다시 쓰는 이주는 피한다.
