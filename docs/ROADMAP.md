# 럭키★오락실 로드맵

> 합의 기준: 2026-07-24 오너 피드백 반영본. 제품 방향 A와 잔불 작전 B를 분리한다.
> 원칙·라이선스·콘텐츠 권리는 [README](../README.md), UI 기준은 [UI-LAYOUT](./UI-LAYOUT.md).

## 제품 방향 A — 현행 정본 (2026-07-24 재확정)

럭키★오락실은 **RisuAI 사용자가 API 응답을 기다리는 동안 들어와 노는 봇카드 세계관 오락실**이다.
특정 IP 게임 하나도, 모든 카드를 억지로 같은 게임으로 바꾸는 자동 생성기도 아니다.

- 운영자가 사용 가능한 봇카드를 선정하고 파서로 인물·로어·에셋·명시 규칙을 회수해 내장 콘텐츠팩으로 만든다.
- 사용자는 봇카드를 직접 준비하지 않아도 내장 캐비닛을 즉시 실행할 수 있다.
- 개인 카드를 직접 넣어 여는 게임도 별도 통로로 유지한다. 이것은 필수 입구가 아니라 확장 기능이다.
- 게임은 짧을 필요가 없다. 응답이 도착했을 때 즉시 떠나고, 나중에 한 번의 선택으로 이어 할 수 있어야 한다.
- 한 콘텐츠팩은 월드컵·퍼즐·액션·장기 게임 등 여러 캐비닛에 재사용될 수 있다.
- 제품 성공은 특정 캐비닛의 완성도뿐 아니라 `즉시 시작 → 중단 보존 → 이어하기 → 다른 세계 발견` 흐름으로 판단한다.

### A와 B의 관계

- **A — 럭키★오락실**: 여러 봇카드 세계와 여러 플레이 길이를 담는 제품·플랫폼.
- **B — 소녀전선: 잔불 작전**: A 안에 놓인 첫 대형 캐비닛이자 콘텐츠 컴파일·결정론·LittleJS·저장 복구의 기술 검증작.
- B의 완료를 A의 제품 검증 완료로 간주하지 않는다. B 확장은 A의 오락실 흐름을 검증한 뒤 판단한다.
- B는 삭제하지 않는다. 현 상태로 유지·디버깅하되 24명·기지 경영·장기 메타 확장은 동결한다.

상세 맥락은 [PRODUCT-MEMORY](./PRODUCT-MEMORY.md), 실행 순서는 [BACKLOG](./BACKLOG.md)를 정본으로 삼는다.

## 구조

```
봇카드
  → 카드 읽기 어댑터 (simbot에서 순수 함수만 선별 이식, 출처 기록)
  → 오락실 공용 데이터 (로어 그래프 · NPC/스프라이트 그룹 · 경제표 · 카드 지문)
  → 재료 검사기 (캐비닛 활성화 판정 + 적합도 리포트)
  → 소형 판정 코어 (시드 난수 · 입력 기록 · 결과 해시 · 게임 버전)
  → 캐비닛들 (표현 자유: shadcn/React 기본, 캐비닛별 지연 로딩)
```

- 모든 추출 결과는 `{ value, source, confidence, evidence, derived }` 공통 계약을 따른다.
- 판정·점수·정답은 항상 판정 코어가 소유한다. 렌더링 라이브러리는 표현·입력만 담당한다.

## 모듈화·무이주 계약 (2026-07-23 오너 조건 — ★강제)

패키지 구조와 의존 방향을 처음부터 다음으로 고정한다. 목적: 나중에 갈아엎는 이주(JS→TS,
저장소 교체, 프레임워크 이주)가 구조적으로 불가능하거나 국소 수술로 끝나게 한다.

```
packages/contracts    버전 스키마·타입·마이그레이션 (도메인 의존 0)
packages/card-io      BinarySource 기반 카드 파싱·에셋 지연 색인
packages/extract      카드→놀이 재료 (로어·NPC·경제표·적합도)
packages/engine       소형 판정 코어 (시드·입력·해시, DOM 금지)
packages/persistence  행동 WAL·스냅샷·에셋 캐시 포트
packages/cabinet-sdk  캐비닛 매니페스트·생명주기 계약
packages/ui           shadcn·디자인 토큰·공용 React 부품
cabinets/*/core       게임별 순수 리듀서 (DOM 금지)
cabinets/*/react      게임별 화면·입력·선택형 렌더러
apps/web              Vite React 앱 — 선언형 캐비닛 조립
apps/report-cli       적합도 측정기 (extract 재사용, UI 무관)
```

1. **의존 방향 단방향**: card-io→contracts, extract→card-io/contracts,
   cabinet core→contracts/engine/cabinet-sdk, cabinet react→자기 core/ui/React, apps→모든 조립 계층.
   캐비닛 core는 extract 구현을 호출하지 않고 계약 데이터만 받는다. React/DOM import는
   `packages/ui`, `cabinets/*/react`, `apps/web` 밖에서 금지한다. 위반은 CI 실패한다.
2. **TypeScript strict를 첫 커밋부터**, 예외 없음. "타입만 .ts, 로직은 .js" 금지.
3. **코어 테스트는 Node에서 브라우저 없이 실행** — cabinet core/engine/extract에 jsdom 의존이 생기면 설계 위반.
4. **저장은 용도별 포트 뒤에 둔다** — `ActionLogStore`, `SnapshotStore`, `AssetCache`,
   `CardSourceStore`를 분리한다. 큰 스냅샷·이미지 작업이 작은 행동 영수증을 막지 않아야 한다.
5. **캐비닛 추가 = 독립 폴더 + 선언형 매니페스트** — 앱 셸에 장르별 조건문이나 전용 배선을
   추가하지 않는다. 빌드 등록은 매니페스트 또는 자동 생성 등록부만 사용한다.
   캐비닛 계약: 필요 재료·최소 데이터 수·요구 신뢰도·시드 규칙·입력 기록·점수 계산·
   공유 가능 정보·카드 원문 공개 범위 선언.
6. **외부 게임·렌더링·물리 라이브러리는 해당 캐비닛의 `react` 계층 + dynamic import로 격리.**
   공용 판정 코어는 외부 엔진 타입에 의존하지 않으며, 엔진을 교체해도 카트리지·시드·입력 기록·결과가 유지돼야 한다.
7. 패키지 간 참조는 workspace protocol + `exports` 공개 API만. 내부 파일 직접 import 금지.

## 성능·저장 계약 (럭키 시뮬레이터 장기 회차 교훈 — ★강제)

- 카드 파일 공용 입구는 `BinarySource { size, read(offset,length) }`다. 전체 파일을 공용 상태의
  거대한 `Uint8Array`로 보관하지 않는다. 에셋은 메타데이터와 지연 참조를 우선 반환한다.
- 카드 해제·그래프 분석·썸네일 생성은 Worker 경계를 전제로 JSON/구조화 복제 가능한 DTO만 사용한다.
  게임 판정 전체 Worker 이전은 실측이 8~10ms 관문을 넘을 때만 시행한다.
- 행동은 공용 직렬 대기열에서 판정하고 소형 WAL을 먼저 확정한다. 전체 스냅샷·통계·캐시 정리는
  유휴 차선에서 병합하며, 과거 전체를 매 행동마다 직렬화·재해시하지 않는다.
- 목록은 최대 192px WebP 썸네일을 사용한다. 화면 근처만 로딩하고 낡은 요청은 취소한다.
  Object URL·메모리·영구 캐시는 모두 상한과 퇴출 시 해제 규칙을 가진다.
- 완료 관문: 버튼 눌림 표시 p95≤50ms(데스크톱), 준비된 로컬 행동 결과 p95≤100ms,
  50→1,000행동 p95 성장≤1.2배, 50ms 초과 메인 스레드 장기 작업 없음, 대형 목록 전체 DOM 생성 금지.
- 성능 진단은 버튼 페인트·대기열·판정·WAL·상태 반영·결과 페인트·백그라운드 저장을 분리 계측한다.

## 그래프 지표 정의 (합의 확정)

- 최장 단순 경로는 쓰지 않는다(순환 그래프에서 부정확). **SCC 압축 후 DAG 깊이 + 최단거리 지름 +
  최소 입력 횟수 BFS 검증**을 쓴다.
- 그래프는 2층: **잠재 그래프**(구조적 연결 가능성)와 **실행 그래프**(특정 시드·턴에서 실제
  activateLore가 활성화한 관계). 퍼즐 정답 검증은 실행 그래프 재실행으로 한다.
- 정규식 키 로어는 기본 퍼즐에서 제외(사람이 읽을 발굴 단어가 없음), 고급 규칙으로 격리.

[BACKLOG](./BACKLOG.md)를 따른다.

  첫 재미 10~30초, 사건 20~60초, 한 판 3~5분. 15분 플레이는 사용자가 원할 때만 확장.

  - 럭키★러시: 프레임 폐기, 검증된 사건 아이디어만 아카이브 방 사건 후보로 보존
- **당시 착수 순서(완료)**: ①공용 카드 재료 검사기 브라우저 연결 → ②최애 월드컵 개봉식 →
  ③카드 복구반 → ④실카드 5종 기술 검증 + 실플레이 재미 검증.

  오너 실플레이 결과 카드 복구반의 기술 성립과 별개로 반복 재미는 보류 판정을 받았다.
  복구반 신규 문제 유형과 럭키★아카이브 확장은 멈추고, 다음 수직 시제품은
  [소녀전선: 잔불 작전 SPEC-C1](./SPEC-C1-gfl-ember-tactical-roguelite.md)로 전환한다.
- **검증 2단계 분리**: 기술 검증(NPC 그룹화 정확도·정답 생성 건전성·관문 차단 정확도) /
  재미 검증(설명 없이 시작, 재도전, 다른 카드 투입 — 실제 행동으로 측정. 아카라이브 게시 반응은
  확산성 지표로만 사용)
- **결과 화면 원칙**: 점수보다 인물 먼저 ("가장 많이 복원한 인물", "우승 최애").
- **순화 번역 원칙**: 방치 성장→탐험 기록 자동 정리, 강화→업적 기반 복원 등급(운 아님),
  뽑기→플레이로 발견한 것의 개봉 연출(중복 없는 확정 신규), 희귀도→카드 내부에서 실제로 드문
  에셋·고립 로어(earned rarity), 일일 과제→고정 시드 도전(접속 강요 없음).
- **리텐션**: "새 카드를 받으면 넣어본다"는 검증된 구조가 아니라 **잠재 경로(가설)**로 취급하고
  재방문 시 신규 카드 투입 비율을 실측한다.

## 공유 계약

## 외부 게임엔진·장르 부품 조달 (2026-07-24 재조사)

### 채택 순서와 관문

1. 기본 UI 게임은 계속 React/CSS로 만든다.
2. 공용 `ArcadeRuntimeAdapter`와 네 엔진 비교 실험은 완료했다. 엔진은 캐비닛 진입 시에만 내려받고 전환·이탈 시 입력·오디오·프레임 루프·GPU 자원을 모두 해제한다.
3. 초기 JS gzip 200KiB, 버튼 표시 p95 50ms, 50ms 초과 장기 작업 없음 기준을 그대로 적용한다. 외부 엔진 청크는 초기 청크에 합쳐지면 실패다.
4. 로그라이크·물리·대형 그래프처럼 실제 장르 요구가 생길 때만 rot.js·Matter.js·Cytoscape.js 중 해당 부품 하나를 추가한다.
5. 엔진 하나를 전역 표준으로 선고하지 않는다. 작은 액션·대형 맵·타일맵 등 실제 게임 수직 시제품마다 비교 보고서와 개발 복잡도를 함께 보고 선택한다.
6. 물리 시뮬레이션은 기기 간 부동소수점 차이가 있으므로 순위·보상 결과를 직접 소유하지 않는다. 고정 스텝을 쓰고 판정 코어의 선결정 결과를 표현하는 것을 기본으로 한다.
7. 첫 엔진 시제품의 목적은 공용 `ArcadeRuntimeAdapter`의 `mount/pause/resume/destroy`, 에셋 지연 공급, 모바일 입력, 오디오 해제, 결정론 리플레이 경계를 검증하는 것이다.

### 네 엔진 베이크오프 결과 — 럭키★더비 실험장 (2026-07-24)

- 소녀전선 콘텐츠팩에서 8명의 초상과 명시 능력치를 뽑아 `DerbyRoster`를 만들고, 30Hz·900틱 정수 연산 경주를 순수 코어에서 먼저 확정한다.
- 같은 시드의 91개 프레임·순위·결과 해시는 네 렌더러에 동일하게 전달된다. 엔진은 선수 위치와 초상만 그리며 승패를 바꾸지 못한다.
- 공용 `ArcadeRuntimeAdapter`를 `cabinet-sdk`로 승격하고 `mount/preload/play/pause/resume/resize/diagnostics/destroy` 수명주기를 고정했다.
- Phaser 358.2KiB gzip, melonJS 180.3KiB, Excalibur 143.2KiB, LittleJS 47.4KiB는 모두 별도 지연 청크다. 초기 로비는 105.7KiB/200KiB를 유지한다.
- 화면의 `4엔진 자동 비교`는 로드·첫 화면·최장 프레임·완주 여부를 로컬에 기록하고 현재 기기 추천 순위를 계산한다.
- 데스크톱·모바일에서 네 엔진 연속 완주, 전환 뒤 Canvas 누적 방지, 기존 캐비닛 회귀를 브라우저 검사로 통과했다.
- 이 실험장은 최종 경마 게임이 아니다. 베팅·공용 재화·장기 육성은 오너가 실제 기기 결과와 화면 손맛을 확인한 뒤 별도 게임 규칙으로 설계한다.

### 첫 엔진 시제품 결과 — 잔불 작전 C1 (2026-07-24)

- 12명과 보스의 허용 표정만 뽑은 내장팩 117파일·9.51MB를 생성했다. 원본 대형 모듈은 저장소와 런타임에 포함하지 않는다.
- 로비 초기 JS gzip은 103.6KiB/200KiB다. LittleJS는 47.4KiB 지연 청크로 전투 진입 시에만 내려받는다.
- 판정 코어가 8라운드 `BattleTranscript`와 결과 해시를 먼저 만들고 LittleJS는 재생만 한다. 1×/2×/4× 모두 결과가 같다.
- 500시드 밸런스 하니스에서 안전 경로 462/500(92.4%), 공격적 경로 429/500(85.8%) 완주를 확인했다.
- 데스크톱·모바일 E2E에서 편성→경로→전투→보상→새로고침 복구를 통과했다. 다음 확장은 오너 실플레이에서 손맛 문제가 확인된 부분만 조정한다.

### 제품 방향 A 구현 결과 (2026-07-24)

- 로비 첫 행동을 카드 업로드에서 `지금 바로 놀기`와 `이어하기`로 바꿨다. 개인 카드 분석은 선택 구역으로 이동했다.
- 캐비닛 매니페스트가 즉시/반복/긴 게임, 내장/카드 입구, 예상 시간, 이어하기 문구를 선언한다.
- `RecentPlay`는 게임 상태를 복제하지 않고 캐비닛 스냅샷 위치와 진행 문구만 저장한다.
- 소녀전선 콘텐츠팩을 공용 `BuiltInContentPack`으로 어댑트해 월드컵·작전 암호 기억·잔불 작전이 공유한다.
- 새 `작전 암호 기억`은 5라운드 결정론 순서 기억 게임이며 1~2분 캐비닛 역할을 맡는다.
- 두 번째 콘텐츠팩 검증 명령은 인물 수·초상 보유·최소 표정 수를 검사해 월드컵과 기억 게임 개방 여부를 출력한다.
- 초기 JS gzip 105.0KiB/200KiB, 전체 브라우저 회귀 12통과·2조건부 건너뜀. 새 캐비닛은 지연 청크다.

## 두 번째 대표 세계 D — 《테메로세: 여백》 (2026-07-24 승인)

테메로세 시리즈 4종을 전수 조사한 결과, 단순 월드컵 온보딩이 아니라 대사 중심 JRPG 수직 시제품을 만들 가치가
확인됐다. 이후 커뮤니티 인기투표에서 Bestiaization 36표(35%), Finale 14표(13.5%)가 확인됐고,
박니은 19표·페일 10표·네모와 카노 각 8표가 캐릭터 상위권을 차지했다. 투표는 장르 선호 조사가 아니므로
덱빌딩 JRPG 구조는 유지하되, 첫인상과 등장 우선순위를 바꾼다. Bestiaization의 피쿼드·메트로·인기 인물을
전면에 놓고 Finale의 여백·부활·생태·규칙형 보스를 게임 뼈대로 사용한다.

플레이어 페르소나는 `항해사`로 확정한다. 미등록 생존자가 고장 난 피쿼드 계약을 직접 깨우고, 임시 항해사로 첫 원정을
지휘한 뒤 여백에서 죽은 항로의 선택을 기억해 돌아오며 직책을 획득한다. 외모·이름·성별과 최종 사상은 비워 두되,
타인을 소모품으로만 취급하지 않고 자기 결정의 결과를 떠넘기지 않는 윤리선은 고정한다.

- 장르: 일러스트 기반 탐험형 덱빌딩 JRPG + 로그라이트.
- 차별점: 런타임 LLM 없이 검수된 대사집이 현재 상태·선택·동료 조합·사망 기억에 반응한다.
- 첫 범위: 죽은 단말기 → 박니은 연결 → 임시 항해사 등록 → 피쿼드 폐허 7노드 → 트레인헤드 → 여백 → 항해사 귀환.
- 등장: 박니은·알제·라일라 기록·워어즈, 동료 페일·카노·네모/바치칼 중 둘.
- 표현: React 대화·덱·지도 + Phaser 4 전투·컷인·왜곡. 판정과 대사 선택은 순수 TypeScript.

### 전체 캠페인 5막

1. **마지막 계약**: Bestiaization의 피쿼드 폐허에서 항해사 직책을 획득한다.
2. **여섯 개의 잃어버린 방위**: 서곡의 유물 항해와 페일 기원, 라일라의 인간 생존론을 만난다.
3. **감정을 되찾은 도시**: 루트2를 공존이 실제로 가능했던 불완전한 가능 세계로 탐험한다.
4. **사백 년짜리 정답**: Bestiaization의 체제와 니은·알제·네모의 실패, 워어즈의 세계 반복을 밝힌다.
5. **다음 시대의 항로**: Finale의 세 층과 세력이 함께 감당할 조건을 모아 대멸종의 예측을 깬다.

중심 질문은 `감정에서 태어난 베스티아를 인간에게 돌려보내는 것은 구원인가 죽음인가`이며, 더 깊은 질문은
`세계를 구할 힘이 있는 사람에게 그 세계의 답까지 정할 권리가 있는가`다. D1 통과 전에는 2~5막을 구현하지 않는다.

### D1 선행 기술 관문

테메로세 카드들은 이미지 메타데이터와 실제 페이로드 형식이 불일치한다. 《Finale》의 941개 이미지는 실제로
모두 WebP지만 현재 메타데이터 경로는 PNG로 해석돼 `asset_mime_signature_mismatch`가 발생한다.
콘텐츠 투입 전에 실제 바이트 시그니처 판별, 장면별 허용 목록, 파생 썸네일, 성인 에셋 차단을 먼저 구현한다.
원본 전체를 저장소나 초기 번들에 넣지 않는다.

### D1 구현 순서

1. D1-0 실제 MIME 스니핑과 안전한 콘텐츠팩 컴파일.
2. D1-1 항해사 단계·첫 행동·잔향·이야기 기억·대사 큐·반복 방지·관계 단계 계약.
3. D1-2 7노드 원정·항해사/동료 덱·인간성·짐승성·잔향·트레인헤드 판정 코어.
4. D1-3 React 대화·선택·기록·중단 저장 화면.
5. D1-4 Phaser 장면·컷인·보스 왜곡 표현.
6. D1-5 세 동료 조합과 승리/사망/재개의 통합 실플레이.

박니은은 Finale 현재 세계에서 사상 지평을 유지해 대멸종을 막고 있으므로 현장 파티원으로 이동시키지 않는다.
원격 임무 주도자이자 첫 얼굴로 배치한다. 키게니아는 인기 공동 4위지만 현재 4개 카드에 페르소나 원본이 없어
구작 카드 확보 전까지 집필·등장을 금지한다.

서곡 원본에서 페일은 `{{user}}`의 죽은 연인이 남긴 감정에서 태어났다. 게임은 이 사실을 2막의 핵심 미스터리로 회수하지만,
현재 항해사에게 과거의 연애나 호감을 자동 지급하지 않는다. 동일 영혼·역할 계승·겹친 항로 중 무엇인지는 플레이어가 종막에서
해석한다.

기존 A의 오락실 입구·이어하기·콘텐츠팩·캐비닛 레지스트리 원칙은 그대로 적용한다. D는 A를 대체하는 별도 앱이
아니라 A 안의 두 번째 깊은 캐비닛이다.
